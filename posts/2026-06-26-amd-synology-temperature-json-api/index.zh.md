# 给 AMD 群晖做一个"可信"的温度接口：CPU、主板、每块硬盘，一份 JSON 喂 ESP32

## 开头

我想要一件很小、很无聊的事：在桌上摆个小表盘，看我那台 NAS 的温度——CPU、主板、每块硬盘。结果在 AMD 的群晖上，这事变成了一条意外深的迷宫：**没有任何一个工具能把所有温度读全**，而能读出*一部分*的几个工具，彼此还对不上。

于是我做了个最小的东西：每个值都从最靠谱的来源读、合并成一份小小的 JSON 接口——然后它**不信任自己**，每次运行都拿独立来源把每一项交叉核对一遍。

这篇讲为什么它难，以及"靠核对、不靠感觉"的监控到底长什么样。

---

## 1. 为什么在 AMD 群晖上读温度这么难

网上每篇"读 NAS 温度"的教程都默认你是 Intel 机器、有一个听话的工具。AMD 的 DSM 机器一次性打破三个假设：

- **自带的 `smartctl` 又老又残**：没有 `--json`，`--scan` 是坏的，必须显式 `-d sat` 才能读到 ATA SMART 属性表。漏了它，你的硬盘看起来就"没有温度"。
- **AMD 不报 "Core 0"**：在 `k10temp` 驱动下，CPU 显示成 `Tctl` / `Tdie`，不是每篇 Intel 教程让你 grep 的 `Core 0` / `Package id 0`。
- **Glances 看得到 CPU、看不到主板**：在 Docker 里跑 Glances 能拿到 CPU 传感器，但这些机器上它就是不吐主板/系统温度，而且容器里读硬盘那条路不稳。

每个工具都缺一块、缺的还不是同一块。这正是这个项目存在的理由：没有单一来源是完整的，只能自己拼一个。

## 2. 思路：每个值挑对来源，合并成一份 JSON

与其逼一个工具干所有事，不如让每个值都来自最可靠地报它的那个来源：

| 值 | 来源 | 怎么读 |
| --- | --- | --- |
| CPU | Glances REST API | `…/api/4/sensors`，标签 `Tctl`（回退 `Tdie`） |
| 系统 / 主板 | 群晖自己的 `synowebapi` | `SYNO.Core.System` → `sys_temp`（就是 DSM 界面显示那个值） |
| 硬盘 | `smartctl` | `-A -d sat /dev/sataN`，属性 `194`（或 `190`） |

一个小 Python 采集器查这三个来源、统一成整数摄氏度、写出一份扁平 JSON。某个源不可用时，对应字段是 `null`，而不是编一个数。没有数据库、没有 agent，只有 shell + Python 3 和两个小容器。

## 3. 别信任自己的管线——每次运行都给自己打分

这是我最得意的一部分。大多数 homelab 监控只是**自洽**：它报它那一个工具说的值，然后你祈祷它是对的。这个项目**每次运行都拿三个独立来源核对自己**，靠一个 `verify.py`：

- **CPU** —— 端点（Glances）对账 直读 sysfs `/sys/class/hwmon/hwmon0/temp1_input`（`k10temp`，完全绕开 Glances）。
- **系统** —— 端点 对账 `synowebapi` 的 `sys_temp`（DSM 官方值）。
- **硬盘** —— 端点（smartctl）对账 DSM 存储管理器（`SYNO.Storage.CGI.Storage`）。

它留 `3°C` 容差，吸收两次采样之间约 30 秒的时差，打印 PASS/FAIL 表，任一项不一致就以非零退出。重点不是"交叉核对很高级"，而是：**一个你没拿第二来源核对过的温度，只是一个看起来很官方的数字而已。**

## 4. 契约：一份扁平 JSON → 一个物理表盘

下游一切都依赖一个稳定的形状：

```json
{ "ts": 1750900000, "unit": "C", "model": "DS1525+",
  "cpu": 53, "cpu_label": "Tctl", "system": 41,
  "disks": [ { "name": "sata1", "temp": 38 } ] }
```

一个守护进程每 30 秒刷新它，并**原子写入**——先写临时文件，再 `mv`——这样读取方永远不会读到写了一半的文件。一个 `nginx:alpine` 容器在 `:8787` 上只读托管它。从这儿往下：

- 一个 **ESP32 / M5Dial** 桌面小玩意儿 HTTP GET 它，用 ArduinoJson v7 解析，用 `ts` 字段判断数据是否过期。
- 一个自包含的 `web/index.html` 把同一份 JSON 在浏览器里渲染成实时表盘：低于 66°C 绿、66–81°C 琥珀、高于 81°C 红——每 5 秒刷新，`ts` 超过 90 秒就亮个"数据过期"的点。

## 5. 几个踩坑故事

几个值得单独成段的坑，因为它们都实打实地耗了时间：

- **会自杀的脚本**：群晖是 busybox，没有 `pgrep`。`pkill -f temps_daemon.sh` 会匹配到*启动器自己*的命令行（里面就含这个路径），把正在管理守护进程的那个东西也杀了。修法：记 PID 文件、按 PID `kill`，绝不按模式匹配杀。
- **Docker 里 CPU 传感器为空**：容器里的 Glances 返回不了传感器，直到你把宿主机 PID 空间给它（compose 里 `pid: host`）——否则 psutil 看不到宿主机硬件。
- **连不上的 `scp`**：往 DSM 部署时报 `subsystem request failed on channel 0`，因为群晖没启用现代 `scp` 默认走的 SFTP 子系统。`scp -O`（旧版协议）解决。

## 6. 诚实的取舍

这是个锋利的小工具，不是企业级栈，边界要讲清楚：

- **它是按一台机器调的**：硬盘槽位、AMD 的 `Tctl` 标签、`hwmon0` 路径、若干卷路径目前都是写死的。换型号、换盘数、换 Intel 机器就得改代码（把这些抽成 config 在 TODO 里）。
- **目前只有温度**：吞吐、负载、风扇转速、卷容量、UPS——全在路线图上，都还没做。
- **物理 M5Dial 固件还只是愿景**：今天的"表盘"是一个浏览器页面加一张 mockup，真机固件还没写。
- **无鉴权、仅限内网**：端点绑在局域网、没有鉴权——绝不端口转发、绝不隧道暴露公网。任何能进你内网的人都能读到这些数。

但这些都不动摇核心收益：在一个温度散落在三个半残工具里的平台上，它给你一个**真正能信**的接口。
